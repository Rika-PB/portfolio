01 — The Challenge

Operating blind
in a sea of data.

"Every financial decision was a calculated guess. The product existed — but its interface made the problem worse, not better."

consalio is a cloud-based SaaS platform helping companies manage external consultants, project budgets, and vendor invoices in real time. By the time I joined, the platform had a critical UX problem: the data was there, but none of it was legible.

CFOs and project managers were drowning in fragmented spreadsheets, chasing late invoices, and operating with zero real-time visibility into budget vs. actuals. The challenge wasn't a simple UI refresh — it was a structural redesign problem: how to translate massive numerical datasets into interfaces that three very different user types could act on immediately.

CFO / Finance

Strategic KPI overview. Budget vs. forecast at a glance. No noise.

Project Manager

Budget deviation alerts. Vendor tracking. Operational clarity.

Assistant

Fast action queue. Simple navigation. Zero learning curve.

03 — Process

Five phases.
One consistent system.

Research & Audit

Deep analysis of the existing product: drop-off points, friction screens, and how each persona experienced the tool differently. Core finding: raw tables with no visual hierarchy forced users to mentally translate data before they could act on it.

Information Architecture & User Flows

Restructured the entire navigation logic around one insight: each role needed a fundamentally different entry point. A CFO lands on KPI summary. A PM lands on deviation alerts. An Assistant on the action queue. Same data — three lenses.

Design System

Before a single screen, I built the system. Typography scale, spacing tokens, a semantic three-state status system (on-budget / at-risk / over-budget), table components, KPI cards, and chart patterns. Built platform-agnostic from day one.

UI Design & Figma Prototyping

With the system in place, screen design moved fast. Centrepiece: the Live Project Cockpit — replacing table-heavy interfaces with visual KPI layers, budget curves, spend-vs-forecast charts, and traffic-light status indicators.

Frontend Implementation

I implemented UI components directly in Tailwind CSS. Later in the Charon phase, I vibe-coded the sticky note feature end-to-end — sketch → code → live product, independently. No interpretation gap, no delay.

06 — Key Decisions

Where senior thinking actually happens.

Principle 01

"First-Click Value" — 5 clicks to understanding

Every screen governed by one rule: the user must understand their financial position within 5 clicks of logging in. This drove radical decisions about what to surface — and what to hide. Complexity belongs in drill-downs. The surface stays clean.

Principle 02

Visual-first, tables second

The previous interface was table-first. The redesign inverted this: charts and KPI cards on the surface, tables as the detail layer underneath. The data didn't change — its hierarchy did. Users reported faster orientation and fewer translation errors.

Principle 03

Role-based entry points

Rather than one universal dashboard, I designed layered views — same underlying data, three different lenses. A CFO's first screen is not a PM's first screen. This single architectural decision reduced onboarding friction across all user types simultaneously.

08 — Reflection

What I learned
building for data.

"The hardest part of data-dense design is not the visual design — it's the information architecture. The hierarchy decision is the design decision."

Building a design system for a financial SaaS tool taught me that getting the hierarchy right is where senior-level thinking actually happens. Deciding what belongs on the surface and what belongs in a drill-down — designing status logic that works coherently across dozens of components — that's the craft that's invisible to users but defines whether a product works.

It also reinforced my conviction that the designer-developer divide is itself a design problem. By closing that gap — implementing components myself — I eliminated an entire category of misinterpretation and shipping delay. The result wasn't just a better product. It was a better process.

The consalio project is where I stopped thinking of myself as a UX/UI Designer who also knows some code — and started thinking of myself as a Product Designer who owns the entire journey from first sketch to shipped feature.
